
### TL 补充 — 低成本反破解方案(用户提供)

| 步骤 | 机制 |
|------|------|
| 1 | 首次购买→后端返回加密token存本地 |
| 2 | 每次启动→静默联网验证token |
| 3 | 验证失败3次→降级为免费模式 |
| 4 | 所有解锁状态→必须经后端API校验 |

防护效果: 防90%静态破解 | 开发量: 1-2天 | 用户体验: 无感静默

## QA — 测试工程师

### 核心关切

| # | 关切 | 说明 |
|---|------|------|
| 1 | **儿童误操作** | 3岁孩子会乱点、多点、长按、连击。所有按钮需防连击 |
| 2 | **离线降级** | 网络断开时token验证失败→降级免费模式，需确保不丢数据 |
| 3 | **多孩切换** | 切换Profile时数据隔离是否彻底？进度、积分、贴纸是否独立？ |
| 4 | **年龄差异** | 3岁和6岁使用同一游戏，难度自适应是否正确？ |
| 5 | **音频并发** | 快速连续点击多个泡泡导致多个语音同时播放 |

### 关键测试用例

```
GIVEN 3岁儿童快速连击泡泡
WHEN 0.3秒内点击3个不同泡泡
THEN 只响应第一次点击，后续点击静默忽略

GIVEN 网络断开+token验证失败3次
WHEN 用户继续使用
THEN 降级为免费模式(仅前200字可用)，已解锁内容不清除，恢复网络后自动恢复

GIVEN 家长在Profile A设置难度3
WHEN 切换到Profile B(年龄3岁)
THEN 难度自动设为1，进度/积分/贴纸完全独立

GIVEN 中文语音播放中
WHEN 立即点击英文语音按钮
THEN 自动取消中文语音，开始英文语音
```

### 边界条件

- 字库为空(首次安装) → 引导页→种子加载→默认前200字
- token过期+无网络 → 3次静默验证→降级→网络恢复→自动重试
- 存储满(IndexedDB超配额) → 提示清理，不丢进度数据
- 同时登录多设备(同一账号) → 进度不同步提示，以最后操作设备为准

### TL 补充2 — 数据模型(用户提供)

采用 **Event Sourcing(事件溯源)** 替代传统状态存储：

```
操作序列模型:
  用户123:
    - 2026-05-30 08:00: learn("山") [设备A, 离线]
    - 2026-05-30 09:00: learn("水") [设备B, 离线]
    - 2026-05-30 10:00: review("山") [设备A, 联网]

冲突解决:
  1. 同设备: 按时间先后合并
  2. 跨设备: 操作全保留 + 去重(同一字学多次只记一次)
  3. 删除: 软删除标记，不物理删除
```

优势: 天然支持离线、多设备同步、审计追踪、可回溯

## CO — 儿童隐私合规

### 核心关切

| # | 关切 | 说明 |
|---|------|------|
| 1 | **COPPA(美国)** | 13岁以下儿童数据收集需家长可验证同意。不收集就不触发 |
| 2 | **GDPR-Kids(欧盟)** | 16岁以下需监护人同意。数据最小化原则 |
| 3 | **中国未成年人保护法** | 不满14周岁未成年人个人信息需监护人同意 |
| 4 | **数据本地化** | 中国用户数据必须存储在中国境内服务器 |
| 5 | **第三方SDK风险** | 微信支付SDK、Firebase等会收集设备信息，需在隐私政策披露 |

### 合规策略: Zero Data Collection (最小化)

```
儿童端: 零数据收集
  - 不收集姓名、年龄、性别
  - 不收集位置、设备ID
  - 学习数据仅存本地 IndexedDB
  - 无第三方分析SDK(SDK-free)
  - 无广告SDK

家长端(最小化):
  - 邮箱(可选，仅用于同步)
  - 支付信息(仅支付渠道处理，App不存储)
  - 学习报告数据(家长主动分享才上传)
```

## Phase 4: 冲突解决

### 检测到的跨角色冲突

| 冲突 | 角色A | 角色B | 描述 |
|------|-------|-------|------|
| C1 | PM | CD | PM要"可玩性最强"→更多游戏化。CD要求"正向激励+无惩罚"→游戏化有度 |
| C2 | TL | CO | TL要求后端验证token(联网)。CO要求"不上传儿童数据"。token验证是否触发隐私问题？ |
| C3 | PM | TL | PM要求前200字免费+按字包付费。TL指出纯前端门控可破解。成本/安全权衡 |
| C4 | CD | CX | CD建议5-6岁可限时挑战。CX认为限时增加焦虑，违背"零压力"理念 |
| C5 | PM | QA | PM要求多孩模式+多设备。QA指出跨设备同步的冲突风险 |

### 推荐权衡方案

**C1: 游戏化有度 (PM vs CD)**
- 方案A: 游戏外壳+学习内核。游戏机制服务于学习目标而非替代它
- **选A**: 泡泡戳破后展示字卡+朗读(已完成)。闯关解锁条件是"掌握N个字"而非"玩N次"

**C2: token验证隐私合规 (TL vs CO)**
- 方案A: token仅含加密hash不含个人信息→不触发COPPA
- **选A**: token = sha256(userId + purchaseId + salt)，无法反推出用户身份

**C3: 支付安全 (PM vs TL)**
- 方案A: 内容门控在前端+后端token双验证。前200字永久免费无门控
- **选A**: 免费部分零门槛，付费内容才需token验证

**C4: 限时挑战 (CD vs CX)**
- 方案B: 限时可开启/关闭，默认关闭。5-6岁模式家长可选开启
- **选B**: 给家长选择权，不做强制

**C5: 多设备同步 (PM vs QA)**
- 方案A: Event Sourcing 操作序列模型(用户提供)。离线操作记录→联网合并→去重
- **选A**: 天然支持离线+多设备，冲突解决规则明确
